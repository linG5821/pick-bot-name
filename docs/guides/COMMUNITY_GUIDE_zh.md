## 🎯 贡献方式

### 1. 🎲 提交生成规则

生成规则是本项目的核心。分享你的创意！

**优秀规则的特点：**
- 独特且有创意的组合
- 文化上适当的内容
- 权重平衡良好
- 清晰的文档
- 高质量的示例

**流程：**
1. 阅读[规则指南](RULE_GUIDE.md)
2. 创建你的规则 JSON 文件
3. 使用 `npm run validate-rules` 测试
4. 提交 Pull Request
5. 回应反馈

**认可：**
- 你的名字出现在规则文件中
- 列入贡献者名单
- 在发布说明中提及

### 2. 🐛 报告 Bug

发现 bug？帮助我们修复它！

**报告前：**
- 搜索现有 issues
- 检查是否已修复
- 收集重现步骤

**报告格式：**
- 使用 bug 报告模板
- 包含截图
- 描述期望行为 vs 实际行为
- 提供环境详情

### 3. ✨ 建议功能

有想法？我们很乐意听到！

**好的功能请求：**
- 解决真实问题
- 与项目目标一致
- 包含用例
- 考虑实现方式

**流程：**
1. 打开功能请求 issue
2. 描述问题
3. 提出解决方案
4. 与社区讨论

### 4. 📚 改进文档

文档总是受欢迎的！

**可改进的领域：**
- 教程和指南
- API 文档
- 代码注释
- 翻译修正
- 示例

### 5. 🌍 添加翻译

帮助项目走向世界！

**翻译需求：**
- UI 文本（zh、en、ja）
- 文档
- 规则描述
- 错误信息

### 6. 💻 代码贡献

准备好深入代码了？

**适合新手的 issues：**
- 寻找 `good-first-issue` 标签
- UI 改进
- Bug 修复
- 测试覆盖率

**开发环境设置：**
```bash
npm install
npm run dev
npm run validate-rules
npm run build
```

## 🏆 认可系统

### 贡献者等级

**🌱 新贡献者**
- 第一个贡献被合并
- 欢迎信息
- 添加到贡献者列表

**⭐ 维护者**

## ❓ 常见问题

**Q: 我是开源新手。从哪里开始？**
A: 从文档开始或提交一个简单的规则。查看 `good-first-issue` 标签。

**Q: 我的 PR 多久会被审查？**
A: 通常在 1-3 天内。自动检查会立即运行。

**Q: 我可以同时处理多件事吗？**
A: 可以，但一次专注于一个 PR 以获得更快的审查。

**Q: 如果我的 PR 被拒绝怎么办？**
A: 从反馈中学习并再次尝试。拒绝是学习的一部分！

**Q: 如何成为维护者？**
A: 随时间推移持续、高质量的贡献。展示领导力并帮助他人。
